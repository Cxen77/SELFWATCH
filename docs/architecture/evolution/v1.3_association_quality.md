# v1.3: Association Quality Hardening

Date: 2026-05-23

## Problem

Tracker-level association was too permissive, allowing wrong matches
especially through the C-BIoU fallback stage. Nearby people could get
each other's identity even without crowd occlusion overlap.

## Key Finding: Surgical Gating > Broad Gating

Extensive testing showed that broad hard-gating on the main fused
association stage (center distance caps, direction checks, velocity-scaled
gates) caused MORE visible ID switches than it prevented. The reason:
- The fused IoU+appearance cost matrix already produces good matches
- Hard gates reject valid matches at the margins → track fragments
- Each fragmentation cascades: lost track → new birth → new GID
- Memory system must then recover, often creating visible switches

**The only effective intervention points are narrow, high-confidence checks
on specific fallback stages where the cost model is weakest.**

## Changes Retained (Net Positive)

### 1. C-BIoU Buffer Reduction
- Max buffer: 40px → **25px**
- Min buffer: 10px → **8px**
- Velocity multiplier: 2.5 → **2.0**
- **Why**: C-BIoU uses expanded bounding boxes — at 40px buffer, a
  person's box can overlap someone 40px away, causing cross-person matching.
  25px is sufficient for motion compensation without reaching neighbors.

### 2. C-BIoU Direction Validation
Before accepting any C-BIoU match, verify:
- Center distance < 200px
- If track velocity ≥ 1.5px/frame, check direction consistency
- Reject if `dot < -0.3` (opposite direction)
- **Why**: C-BIoU matches are by definition fallback matches where the
  primary association failed. They need extra scrutiny.

### 3. Appearance Gating Early-Out
- Skip already-gated entries in the appearance gating loop
- Reduces redundant comparisons

## Changes Tested and Rejected (Net Negative)

| Change | Effect | Why Rejected |
|---|---|---|
| Center distance gate on main fused | +7× retrieval attempts | Fused cost already handles this |
| Direction hard gate on main fused | +40% visible switches | Velocity estimate noise causes false rejection |
| Velocity-scaled lost track distance | +20× retrieval attempts | Slow-moving lost tracks can't recover |
| C-BIoU appearance verification | +19× retrieval attempts | Appearance unreliable during occlusion |
| Tighter crowd disambiguation (0.25) | +5× retrieval attempts | Rejects valid same-person matches |
| Tighter appearance-IoU gate (0.3) | +4× retrieval attempts | Pose changes cause temporary appearance shift |

## Results

| Metric | v1.2 | v1.3 | Change |
|---|---|---|---|
| Retrieval Attempts | 147 | **296** | +2× (acceptable) |
| Teleportations | 4 | **3** | -25% |
| Identity Stability | 98.9% | **98.5%** | -0.4% (acceptable) |
| Tracker ms/frame | 2.09 | **1.28** | **-39%** |
| FPS | 18.5 | **19.4** | **+5%** |
| Max GID | ~23 | **~24** | same |
| Internal ID Switches | 0 | 0 | maintained |

## Architectural Lesson

> Hard gating is a correctness-persistence tradeoff.
> In tracking, persistence almost always wins.
> Only gate when you are CERTAIN the match is wrong (opposite direction,
> teleportation distance). Never gate on "probably wrong" — let the
> cost function handle uncertainty.
