# v1.4: Ambiguity-Aware Identity Freezing

Date: 2026-05-23

## Problem
During scene ambiguity, noisy/weak matches, or brief occlusions, the tracker was spawning replacement local tracks aggressively nearby before the original track could be recovered. Furthermore, provisional identities were automatically committed and rendered without a verification gate, leading to visual ID switches.

## Key Solutions

### 1. Ambiguity-Aware Identity Freezing
- **Trigger**: Freezes identities that have:
  - Fallen back to **C-BIoU** matching.
  - A high fused association cost (`cost > 0.65`).
  - A provisional state with low detection score (`score < 0.60`).
- **Effect**: Merges these GIDs into `frozen_gids`. This immediately:
  - Locks their state as `VISIBLE_FROZEN` in the Central State Machine.
  - Suspends any reassignments/proposals for these GIDs.
  - Suspends appearance embedding template updates (preventing template pollution).

### 2. Birth Suppression near Lost Tracks
- **Trigger**: Track-Aware NMS is extended to check both active tracks and recently lost/frozen tracks (`time_since_update <= 45` frames).
- **Effect**: If a new detection overlaps with a lost/frozen track's predicted trajectory box (`IoU > 0.20`), the creation of a new local track is suppressed. This prioritizes the recovery of the existing identity instead of aggressively spawning a replacement identity.

### 3. Strict Quality-Gated Provisional Commitment
- **Trigger**: Provisional tracks are no longer promoted automatically by frame count alone.
- **Verification Gate**: Promotion to a committed/rendered identity now requires:
  - Stably tracked consecutive hits (`consecutive_hits >= COMMIT_DELAY`).
  - Decent detection confidence (`score >= 0.55`).
  - Physically plausible velocity profile.
- **Effect**: Prevents transient/noisy detections from spawning and rendering new GIDs prematurely.
